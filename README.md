# Code Review: ProcessIncomingCallJob

> Ревью задачи обработки входящего звонка в Laravel.
> Оригинальный код — [`src/Original/ProcessIncomingCallJob.php`](src/Original/ProcessIncomingCallJob.php).

### Краткий глоссарий

| Термин | Значение |
|---|---|
| **Job** | Задача в очереди (Redis). Воркер берёт её, выполняет `handle()`, при ошибке — retry. |
| **Worker** | Процесс `queue:work`, который берёт Jobs из очереди и выполняет. Обычно запущено несколько параллельно. |
| **Race condition** | Ошибка, когда два параллельных worker-а одновременно читают и меняют одни и те же данные, получая некорректный результат. |
| **RPS** | Requests Per Second — запросов в секунду. |
| **Telephony RPS** | Количество HTTP-запросов в секунду во внешнюю телефонию. Равно количеству входящих звонков (1 звонок = 1 запрос). |
| **Lock wait** | Время ожидания, пока другой worker освободит блокировку на строку в БД. |
| **Connection pool** | Пул соединений к БД (PgBouncer, ProxySQL). Позволяет 100 workers работать через 20–30 реальных подключений. |
| **Backoff** | Задержка перед повторной попыткой. `$backoff = [5, 15, 30]` означает: первый retry через 5 сек, второй через 15, третий через 30. |
| **Idempotency key** | Уникальный идентификатор запроса. Если отправить два запроса с одним ключом, сервер обработает только первый. |
| **DLQ** | Dead Letter Queue — очередь для Jobs, которые не удалось обработать после всех попыток. |

## Допущения и предположения

Поскольку в задании не описан ряд аспектов, зафиксирую предположения:

| Область | Предположение |
|---|---|
| **Очередь** | Redis-драйвер, `queue:work` запущен в нескольких процессах |
| **БД** | MySQL / PostgreSQL с поддержкой `SELECT ... FOR UPDATE` |
| **Телефония** | HTTP API с неизвестными гарантиями (может быть медленной, может падать, может дублировать) |
| **Модель Call** | Статусы: `new` → `assigned` → `in_progress` → `completed`; возможен `pending_operator`, `failed` |
| **Модель Operator** | Флаг `available` + `last_call_at` для round-robin распределения |
| **Legacy** | В production уже могут быть данные с непоследовательными статусами |

### Риски из-за неопределённости

1. **Телефония может принимать повторные запросы как новые звонки** → нужен idempotency key, иначе оператору придёт два звонка.
2. **Redis может быть недоступен** → Job потеряется или зависнет. Redis хранит очередь в оперативной памяти — если процесс падает, tasks, которые не завершились, теряются. Нужен DLQ (Dead Letter Queue) для хранения неудачных задач и мониторинг, чтобы узнать о проблеме.
3. **Поведение `save()` при конкурентной записи** — нет optimistic locking (column `version` в таблице). Без него два воркера могут одновременно прочитать одну строку, и второй `save()` перезапишет изменения первого. Это называется lost update.
4. **Нет чёткого SLA телефонии** — таймаут HTTP может быть 1 секунда, а может 30. От этого зависит `$timeout` Job: если таймаут телефонии 30 сек, а `$timeout` Job 10 сек — Laravel убьёт Job, telecom request не успеет завершиться, и при retry телефония получит дубль.
5. **Неизвестно, кто освобождает оператора** — в текущем коде `available = false` ставится при назначении, но нигде не видно, кто возвращает `available = true`. Если процесс упадёт и никто не освободит оператора — он останется «занятым» навсегда.

## Найденные проблемы

### 🔴 Критические

#### 1. Race condition: два воркера захватывают одного оператора

```
Worker A: $operator = Operator::where('available', true)->first();  // Operator #1
Worker B: $operator = Operator::where('available', true)->first();  // Operator #1 (тот же!)
Worker A: $operator->available = false; $operator->save();
Worker B: $operator->available = false; $operator->save();          // перезапись
```

Оба воркера назначают **одного и того же** оператора на два разных звонка.

**Исправление:** Атомарный захват через `UPDATE ... WHERE available = true ORDER BY last_call_at LIMIT 1` внутри транзакции, либо `SELECT ... FOR UPDATE`.

См. [`src/Fixed/ProcessIncomingCallJob.php`](src/Fixed/ProcessIncomingCallJob.php) → метод `acquireOperator()`.

---

#### 2. Отсутствие транзакции

Чтение `Call`, обновление `Operator`, запись `Call` — три отдельные операции. Если процесс упадёт после `$operator->save()`, но до `$call->save()`:
- Оператор уже `available = false`
- Звонок остаётся `status = new` без оператора
- При повторе Job (retry) оператор не будет найден (он «занят»)

Транзакция гарантирует атомарность: либо все три операции успешно завершаются, либо ни одна. Если что-то падает посередине — БД автоматически откатывает все изменения (rollback).

**Исправление:** Обернуть всю бизнес-логику в `DB::transaction()`.

---

#### 3. HTTP-вызов в телефонию без обработки ошибок

Если `sendCallAssigned()` бросает исключение:
- Job падает и retry-ится (OK).
- Но на повторе `$call->status === 'new'` — **нет, он уже `assigned`** после первого запуска (если save прошёл). Это может работать как идемпотентность, а может нет — зависит от того, успел ли `save()`.

Если `sendCallAssigned()` висит 30 секунд:
- Worker заблокирован — он не может брать другие Jobs из очереди.
- `$timeout` Job не задан — Laravel убьёт процесс только когда истечёт `retry_after` в Redis (по умолчанию 90 сек). Проблема: Redis решит, что Job «потерян», и отдаст его другому worker → два worker-а обработают один и тот же звонок.

**Исправление:**
- Явный `$timeout = 30` на Job — чтобы Laravel убивал процесс через 30 сек, а не ждал 90.
- HTTP-клиент с таймаутом 5-10 сек — чтобы не ждать ответа телефонии вечно.
- try/catch + логирование — чтобы знать, что пошло не так.
- Idempotency key на повторный запрос — чтобы телефония не обработала один звонок дважды.

---

#### 4. Отсутствие `failed()` — при исчерпании попыток звонок «зависает»

Если все 5 попыток провалились (нет операторов → Exception), звонок остаётся `status = new` навсегда. Никто не узнает о проблеме.

**Исправление:** Метод `failed()` переводит звонок в `status = 'failed'` и пишет лог/алерт.

---

### 🟡 Важные

#### 5. `throw new Exception('No available operators')` — это не ошибка, это бизнес-ситуация

Нет свободных операторов — нормальная ситуация в колл-центре. Exception:
- Увеличивает счётчик ошибок в мониторинге (шум).
- Срабатывает retry с backoff, но backoff не задан — повторит сразу.

**Исправление:** Не менять статус — оставить `new`, поставить отложенный retry (`$this->release(10)`), не бросать Exception.

---

#### 6. Нет `$backoff` — мгновенный retry

`$tries = 5`, но `$backoff` не задан. Все 5 попыток могут выполниться за секунды, что бессмысленно для ситуации «нет операторов» — они не появятся через 0.5 сек. Backoff даёт системе время: за 30 сек оператор может освободиться, за 120 сек — наверняка.

**Исправление:** `$backoff = [5, 15, 30, 60, 120]` — экспоненциальная задержка.

---

#### 7. Нет идемпотентности при повторе Job

Если worker упал после `$call->save()` (status = assigned), но до `Log::info`:
- Job повторится (Redis не получил ACK).
- `$call->status === 'new'` → false, Job выйдет — OK в этом конкретном месте.

Но если worker упал **после** `$operator->save()`, но **до** `$call->save()`:
- Оператор занят, звонок `new`.
- Retry → нет оператора → Exception → зависание.

**Исправление:** Транзакция (п.2) + атомарный захват (п.1) решают оба сценария.

---

#### 8. `$callId` — нет типа, нет валидации

`$callId` может быть `null`, строкой, массивом. Если передан `null`, `Call::find(null)` вернёт `null` — тихий пропуск без лога.

**Исправление:** Тип `int` в конструкторе + early return с warning-логом.

---

### 🟢 Было бы хорошо сделать

#### 9. Отсутствие мониторинга и метрик

Нет счётчиков:
- Время от `call.created_at` до `call.assigned_at` (SLA — за сколько секунд звонок назначается оператору).
- Процент звонков без оператора.
- Доля failed jobs.

Без этих метрик невозможно понять, работает ли система нормально или уже нет. Логи не подходят для мониторинга — их нельзя агрегировать и построить график.

**Исправление:** Добавить `assigned_at` в модель Call, писать метрики в StatsD/Prometheus — системы, которые собирают числа и строят графики.

---

#### 10. Жёсткая привязка к `app()` для TelephonyClient

`app(TelephonyClient::class)` — разрешение через контейнер прямо в handle. Проблема: при тестировании мы не можем подставить mock (фейковый объект) — придётся привязывать его в контейнер вручную. Если бы TelephonyClient передавался через конструктор, PHPUnit автоматически подставил бы mock.

**Исправление:** Inject через конструктор или метод.

## Что я НЕ стал бы делать прямо сейчас

1. **Event-sourcing / CQRS** — оверкилл для текущей задачи. Достаточно транзакций.
2. **WebSocket / Real-time пуш оператору** — если телефония уже есть, пушить в неё достаточно.
3. **Микросервисы** — выносить Job в отдельный сервис пока рано; сначала убедимся, что один сервис справляется.
4. **Сложный routing-алгоритм операторов** (навыки, языки, приоритеты) — сначала фиксим базовую корректность.
5. **Оптимистичный locking на Call через `version`** — сейчас достаточно проверки `status === 'new'`; добавлю если появятся lost updates на проде.
6. **Блокировка `clients` таблицы** — привязка клиента идемпотентна и не вызывает race.

## Вопрос про масштабирование (×10–50)

### Ожидаемые bottleneck'и

| Компонент | Текущая проблема | При росте ×50 |
|---|---|---|
| **БД: operators** | `SELECT ... WHERE available ORDER BY last_call_at` с row lock — каждая транзакция блокирует строку | При 500 операторах и 100 RPS — сериализация на уровне таблицы. All workers ждут release lock |
| **БД: calls** | Каждый Job делает 2+ запроса к `calls` + 1 к `clients` | I/O растёт линейно; connection pool (default ~100) исчерпается |
| **Redis queue** | Один queue, один worker group | При росте depth очереди — consumer lag; также `retry_after` может убивать долгие jobs |
| **HTTP telephony** | Синхронный вызов в each Job | RPS в телефонию = RPS звонков. При 500 RPS и timeout 5 сек → 2500 одновременных соединений |
| **Логирование** | `Log::info` на каждый звонок | I/O на диск; при structured logging в ELK — буфер может не успевать |

### Простое увеличение workers: что даст и где перестанет

**Что даст:**
- До ~10–20 workers: пропускная способность растёт почти линейно, если БД на SSD и connection pool достаточный.

**Где перестанет помогать:**
- **~20–30 workers:** БД-соединения исчерпаны (default `max_connections` = 151 в MySQL). Каждый worker держит 1–2 соединения — одно для чтения, второе для транзакции. Получается 40–60 соединений, а нужно ещё место для веб-процессов, cron-задач и т.д.
- **~30+ workers:** Lock contention на `operators` — workers проводят больше времени в ожидании lock, чем в полезной работе. Представьте очередь из 30 человек к одной кассе: каждый ждёт, пока предыдущий закончит. Throughput выходит на плато, latency растёт.
- **~50+ workers:** OOM на сервере, scheduler thrashing — операционная система тратит больше времени на переключение между процессами, чем на саму работу.

### Лимиты по компонентам

#### Redis
- **Queue depth:** при 500 calls/sec и 5 sec обработки → 2500 jobs в очереди — не проблема для Redis. Redis хранит всё в оперативной памяти, миллион ключей занимают ~100 МБ.
- **Проблема:** `retry_after` (default 90 sec). Redis-драйвер очередей Laravel работает так: worker забирает Job из списка, но не удаляет его, а помечает как «обрабатывается». Если прошло `retry_after` секунд, а worker не завершил — Redis считает, что worker умер, и отдаёт Job другому. Если Job реально обрабатывается 30+ секунд (медленный HTTP в телефонию), а `retry_after` = 90 сек — всё ок. Но если HTTP-запрос висит 60+ сек, два worker-а начнут обрабатывать один и тот же звонок.
- **Решение:** Увеличить `retry_after` > `$timeout` (например, `retry_after = 90`, `$timeout = 30` — запас в 3 раза), включить `$tries = 1` + ручной retry, или перейти на Redis 7+ с `XAUTOCLAIM` (новый механизм, который точнее отслеживает «жив» ли worker).

#### БД
- **Connections:** 50 workers × 2 connections = 100 → нужен `max_connections` ≥ 200 + connection pooling (PgBouncer / ProxySQL). PgBouncer работает как «прокси» между workers и БД: 100 workers подключаются к PgBouncer, а PgBouncer держит всего 20–30 реальных соединений с БД и переиспользует их. Это похоже на такси-диспетчерскую: 100 машин не едут напрямую к клиенту, а диспетчер (PgBouncer) распределяет заказы между 20 свободными.
- **Lock contention на `operators`:** При атомарном `UPDATE ... WHERE available ORDER BY` — каждый UPDATE блокирует строку в БД. Пока один worker обновил оператора и транзакция не закоммичена — другие workers ждут. Решение: перейти на Redis-пул (см. ниже), где блокировок нет вообще.
- **Writes:** При 500 RPS → 1500+ writes/sec (call + operator + client). Нормально для SSD, но нужно убедиться, что нет long-running transactions — они блокируют строки и мешают writes.

#### HTTP-интеграция с телефонией
- **Connection pool:** Laravel HTTP client (Guzzle) по умолчанию не переиспользует соединения между Jobs. Каждый запрос — новый TCP-handshake (установка соединения → запрос → разрыв). При 500 RPS → 500 handshake/sec. Это медленно: каждый handshake — 3 дополнительных roundtrip (SYN/SYN-ACK/ACK). Решение: включить `Connection: keep-alive` — тогда одно TCP-соединение переиспользуется для нескольких запросов.
- **Timeout:** Если телефония отвечает 10 сек → 50 workers заблокированы на I/O. Worker стоит и ждёт ответа, не делая полезную работу.
- **Решение:**
  - Асинхронная отправка: Job кладёт событие в отдельную Redis-очередь → TelephonyNotifyJob с concurrency limit. Идея: отделить «назначить оператора» (быстро, критично) от «уведомить телефонию» (медленно, можно retry).
  - HTTP client с `Connection: keep-alive` и pool — чтобы не создавать TCP-соединение на каждый запрос.
  - Circuit breaker: паттерн «автомат» — при 5+ ошибках подряд перестаём слать запросы на 30 сек. Это как автоматический предохранитель: если телефония лежит, не нужно бомбить её запросами, лучше подождать.

#### Логирование
- **Volume:** 500 RPS × 2 log lines = 1000 lines/sec. Файл — OK; ELK/Splunk — может отставать.
- **Решение:** Buffer logging (сначала пишем в буфер в памяти, разом сбрасываем на диск), async log shipper (Fluentd/Filebeat — утилиты, которые следят за файлом логов и асинхронно отправляют новые строки в ELK/Kibana, не блокируя приложение), метрики вместо логов где возможно (вместо «Log::info на каждый звонок» — увеличить счётчик в Prometheus, который дешевле по I/O).

### План масштабирования (поэтапно)

#### Фаза 1: Quick wins (1–2 недели)

Быстрые изменения, которые не требуют архитектурных переделок, но сразу повышают устойчивость.

- [ ] Connection pooling в БД (PgBouncer / ProxySQL) — чтобы 50 workers не исчерпали `max_connections`
- [ ] `$timeout`, `$backoff`, `retry_after` выровнены — чтобы Job не дублировался из-за тайминга Redis
- [ ] HTTP client с keep-alive и timeout 5 сек — чтобы не ждать телефонию вечно
- [ ] Отдельная очередь `calls` с приоритетом — чтобы звонки не стояли в одной очереди с фоновыми задачами
- [ ] Мониторинг: queue depth, job duration, error rate — чтобы узнать о проблеме до того, как пользователи начнут жаловаться

#### Фаза 2: Eliminate DB bottleneck (2–4 недели)

Убираем главную проблему: блокировки в БД при выборе оператора.

- [ ] **Redis-пул операторов** через sorted set + ZPOPMIN (см. [`src/Fixed/OperatorPool.php`](src/Fixed/OperatorPool.php))
  - `ZPOPMIN` — O(log N), нет блокировок в БД. Вместо «заблокировать строку → обновить → разблокировать» делаем «извлечь из sorted set в Redis» — это одна атомарная операция, которая выполняется за микросекунды.
  - Восстановление при crash: periodic sync Redis → DB — если Redis перезагрузился, данные берутся из БД
- [ ] Read replicas для `Client::where('phone')` — реплика БД только для чтения. Поиск клиента по номеру не требует строгой консистентности (данные клиента редко меняются), поэтому можно читать из реплики и не нагружать основную БД.
- [ ] Batch insert логов (или switch на async log driver) — вместо 1000 отдельных записей на диск собираем в буфер и пишем разом

#### Фаза 3: Decouple telephony (1–2 недели)

Отделяем «уведомить телефонию» от «назначить оператора». Зачем: если телефония тупит, воркеры не должны простаивать.

- [ ] Вынести `sendCallAssigned` в отдельную очередь `telephony`:
  ```
  ProcessIncomingCallJob → БД-запись → dispatch(TelephonyNotifyJob)
  ```
  Теперь ProcessIncomingCallJob отрабатывает за ~50ms (только БД) и освобождает worker. TelephonyNotifyJob работает отдельно и может хоть 30 сек ждать ответа.
- [ ] Rate limiter на отправку в телефонию — ограничить до N запросов/сек, если у внешней системы есть лимиты
- [ ] Circuit breaker на TelephonyClient — если телефония лежит, перестаём её дергать на 30 сек
- [ ] Retry с exponential backoff для HTTP-ошибок — 1 сек, 5 сек, 30 сек, 2 мин

#### Фаза 4: Horizontal scaling (по мере роста)

Когда один сервер не справляется — добавляем ещё.

- [ ] Отдельные серверы под worker-ы — workers не хранят состояние (stateless), поэтому их можно добавлять сколько угодно. Вертикальное масштабирование (более мощный сервер) дорого и имеет потолок; горизонтальное (больше серверов) — дешевле и гибче.
- [ ] Redis Cluster для очередей — если один инстанс Redis не справляется с объёмом. Cluster распределяет данные между несколькими узлами.
- [ ] Рассмотреть переход с Redis Queue на **RabbitMQ / Kafka** если:
  - Нужен strict ordering (calls одного клиента должны обрабатываться по порядку) — Redis Queue не гарантирует порядок
  - Нужен DLQ с подтверждением — встроенная функция RabbitMQ, в Redis нужно писать руками
  - Нужна replayability (переиграть все события за последний час) — Kafka хранит историю, Redis — нет

#### Фаза 5: Observability (ongoing)

Наблюдаемость — графики и алерты. Без них масштабирование вслепую: не видно, где узкое место.

- [ ] Dashboards: Grafana + Prometheus
  - `call_assignment_duration_seconds` (histogram) — распределение: 50% звонков назначаются за 200ms, 99% за 1.5 sec
  - `operators_available_count` (gauge) — сколько операторов свободно прямо сейчас
  - `job_failure_rate` (counter) — сколько Jobs упало за последний час
  - `telephony_request_duration_seconds` (histogram) — насколько быстро отвечает телефония
- [ ] Alerts:
  - Queue depth > 1000 — Jobs скапливаются, workers не успевают
  - No available operators > 5 минут — все операторы заняты слишком долго
  - Telephony error rate > 5% — внешняя система барахлит
  - Failed calls > 0 — звонки теряются, нужна реакция

### Сводная таблица масштабирования

Таблица читается по горизонтали: каждая строка — отдельная метрика нагрузки, столбцы — уровни роста. Столбец **«Решение»** указывает, что нужно сделать, когда метрика упирается в лимит.

| Метрика | Сейчас | ×10 | ×50 | Решение |
|---|---|---|---|---|
| Calls/sec | ~10 | ~100 | ~500 | — |
| Workers | 3–5 | 20–30 | 50–100 | Redis pool + conn pool |
| DB connections | ~10 | ~60 | ~200 | PgBouncer |
| Lock wait (operators) | ~0 | ~5ms | ~50ms | Redis pool |
| Telephony RPS | ~10 | ~100 | ~500 | Decouple + rate limit |
| Log volume | ~20/sec | ~200/sec | ~1000/sec | Async + metrics |

#### Что значат строки

- **Calls/sec** — входящих звонков в секунду. Главная метрика, от которой зависят все остальные.
- **Workers** — количество параллельных процессов `queue:work`, обрабатывающих Jobs. Каждый worker держит 1–2 соединения с БД.
- **DB connections** — одновременных подключений к базе данных. MySQL по умолчанию держит `max_connections = 151` — при ~60 соединениях уже нужно connection pooling (PgBouncer / ProxySQL).
- **Lock wait (operators)** — сколько времени воркер ждёт, пока другой воркер освободит блокировку на строку в таблице `operators`. При 50ms+ полезная работа воркера падает — он проводит больше времени в ожидании, чем в обработке.
- **Telephony RPS** (Requests Per Second) — количество HTTP-запросов в секунду во внешнюю телефонию. Один входящий звонок = один запрос `sendCallAssigned()`. Поэтому Telephony RPS равен Calls/sec 1:1. Это опасно, потому что внешняя система не наша: при 500 RPS и таймауте 5 сек будет до 2500 одновременных HTTP-соединений. Если телефония начнёт тормозить — наши воркеры заблокируются. Решение: вынести вызов в отдельную очередь (decouple) и ограничить скорость отправки (rate limit).
- **Log volume** — строк логов в секунду. При 1000/sec обычный файл ещё справится, но ELK/Splunk может начать отставать. Решение: буферизация и метрики вместо логов.
